## 1. Editor with markdown toggle + multi-tab

The editor view needs three distinct modes per file: **raw markdown** (plain text, monospaced, full syntax visible), **rendered preview** (parsed markdown, styled output), and **edit mode** (rendered but contenteditable or a hybrid split). A toggle in the toolbar — ideally a segmented control with three states — handles this per-tab.

The tab bar sits above everything. Each tab holds independent state: which file is open, which mode it's in, scroll position, any unsaved edits. The Echoes tab is the first tab and has no close button — it's the persistent home/chat view of the whole app. All other file tabs are closeable. When you open a file from the graph canvas or the sidebar, it either opens a new tab or focuses an existing one if that file is already open.

On the PyQt/PySide side, tabs can be `QTabWidget` with a custom tab bar that intercepts the close button on all but the first tab. The editor itself is best served by a `QWebEngineView` with a local HTML page running CodeMirror (for raw mode) and a markdown renderer like `marked.js` for preview mode — this gives you proper syntax highlighting and live preview without reinventing the wheel in native widgets.

## 2. Graph canvas (the brain view)

This is a **D3 force-directed graph** rendered inside a `QWebEngineView`. When you click the vault icon, you transition to this full-screen canvas. The canvas has a dark graph-paper or subtle dot-grid background — not black, more like a deep slate — so it reads as a distinct "space" vs. the document view.

The **hierarchy** here is important. Directories are not nodes in the traditional sense — they're **gravity wells or labeled regions** that cluster their children. Think of them as soft containers drawn with a faint rounded boundary and a label, not as clickable boxes. Files are the actual nodes (circles or pills), and the edges between them represent semantic connections.

**Fractal expansion**: the root vault starts collapsed. Clicking a directory causes its children to animate outward from the directory's centroid — a smooth radial bloom with spring physics. This isn't just expanding a tree, it's a spatial emanation. Each child node spawns at the parent's position with zero velocity and a radial impulse, then the force simulation settles them into a stable arrangement. The `d3-force` simulation handles this naturally if you batch-add nodes and re-run the simulation with a short alpha restart. The fade-in is an opacity transition, not a move — so you get the combo of "fades in while settling into position," which reads as very polished.

Files have their connection edges drawn as faint curved lines (cubic bezier, not straight). Edge opacity scales with connection strength so weak connections are nearly invisible and strong ones are clear. Clicking a file node routes to its editor tab.

## 3. The fingerprint system — the core design challenge

This is the most intellectually interesting piece, and also the most important one to get right because it underpins both the graph canvas edges and the LLM's ability to reason about note relationships without blowing up its context window.

The key insight from your description: **you never want to pass full note content to the LLM just to figure out connectivity.** The fingerprint is a compact proxy for content — the LLM reasons about fingerprints, not documents.

Here's how to make it work concisely:

**The fingerprint is a structured 3-part string stored in the YAML frontmatter of each note:**

```yaml
fingerprint: "concepts:[rl-training,reward-shaping,policy-gradient] | domain:[ml,research] | hash:a3f9"
```

Three components: a small set of **semantic concept labels** (5–8 terms, generated by the LLM), a **domain cluster** (2–3 broad categories), and a short **content hash** (first 4 chars of the SHA256 of the note body, used for change detection). The whole thing is under 100 characters.

**When a new note is created**, the pipeline does the following:

1. LLM reads the new note content and generates its concept labels and domain cluster.
2. The system passes to the LLM **only the fingerprint strings** (not content) of all existing notes. At 100 chars per note, 500 notes = 50k chars = ~12k tokens. Manageable, and you can further reduce this by only passing the top-K most potentially-related fingerprints using a fast pre-filter (see below).
3. The LLM returns: the new note's fingerprint, plus a list of note IDs it considers connected, with a brief reason for each connection.
4. These connections are stored as edges in SQLite.

**Pre-filter using vector embeddings to avoid passing all fingerprints**: Run a lightweight local embedding model (e.g. `nomic-embed-text` via Ollama, or `sentence-transformers/all-MiniLM-L6-v2` which is 22MB) on each note's concept label string (not the full content — just the fingerprint concepts). This gives you a 384-dim float vector. Store it as a `BLOB` in SQLite. When indexing a new note, do a cosine similarity search against all stored vectors — this takes milliseconds locally and returns the top-20 candidates. You now only pass those 20 fingerprints to the LLM, not the full 500. This keeps the token count roughly constant regardless of vault size.

**Token scaling behavior**: with this design, the LLM call for a new note is: full note content (~1–3k tokens) + 20 × 100-char fingerprints (~500 tokens) = ~2k–4k tokens total. That does not scale with vault size. That is the goal.

**The recursive/inheritance idea**: when the LLM assigns concepts to a new note, it's encouraged (via system prompt) to prefer existing concept terms from other notes before minting new ones. This means the concept vocabulary converges over time — related notes share exact concept strings, making overlap detection trivial. Over time the concept space becomes a controlled ontology that grows only when genuinely new topics appear.

## 4. The indexing system

This is a SQLite database (`vault.index.db`) that lives in a hidden directory inside the vault root (`.echoes/`). It has three core tables:

**`notes`**: `(id, path, modified_at, content_hash, fingerprint_text, vector_blob, dirty)`  
**`edges`**: `(source_id, target_id, strength, reason, created_at)`  
**`tags`**: `(note_id, tag, source)` — where source is either `"llm"` or `"user"`

The **watchdog** file watcher (`watchdog` library in Python) monitors the vault directory. On any file change it sets `dirty=1` on the affected note and enqueues it for re-indexing. Re-indexing runs in a background thread (or asyncio task) so the UI never blocks. The queue is batched: if you save 10 files rapidly, they're processed together in one LLM call, not 10 separate calls.

**Incremental updates**: when a note is re-indexed, only its outgoing edges are deleted and recomputed. Other notes' edges to it remain until their next re-index cycle. This keeps the index eventually consistent without requiring a full re-scan on every change.

**Initial vault scan**: on first launch, or when the index is missing, a full scan runs in the background with a progress indicator. Notes are embedded first (fast, local model), then the LLM fingerprint assignment is batched in groups of 20–30 notes to amortize API call overhead.

## 5. Graph canvas connection rendering

With the fingerprint system in place, the graph canvas now has meaningful edges. **Connection types** can be visually distinguished:

- **Concept overlap** (two notes share concept labels): thick solid edge, color from the shared concept's domain cluster
- **Vector similarity above threshold** (0.8 cosine sim): thin dashed edge
- **Explicit wikilink** (`[[note-name]]` in content): bright solid edge with a small arrowhead, since these are intentional rather than inferred

In the canvas, nodes can be colored by their primary domain cluster. This means related notes form visual clouds even before you look at the edges — the color grouping is a second visual channel on top of the physical proximity from the force simulation.

**The directory regions** in the canvas are computed as convex hulls around all file nodes belonging to that directory, drawn as faint filled polygons with a label. As you expand directories, these hulls grow and reshape dynamically using the D3 hull geometry utilities.

## The key open question to resolve

The one design tension here is **when to run the LLM fingerprint assignment**. You have three options: on every save (expensive, introduces latency), on manual trigger (user-driven, misses passive notes), or async background after a debounce delay (e.g. 30 seconds after the last edit to a note). The third option is almost certainly correct — it keeps the UX fast, the index stays fresh, and users rarely need graph connections to update in real-time while they're actively writing.

The fingerprint concept vocabulary should also have a max size (maybe 200 unique concepts) and a consolidation step where the LLM periodically merges near-synonyms (`"reinforcement-learning"` and `"rl-training"` collapsing into one). This prevents vocabulary fragmentation over a large vault and keeps the pre-filter effective.
